# Ash Hawk Auto Research Pipeline Spec

## Recommended Updated Pipeline

## 1. Purpose

This spec defines the full multi-role improvement pipeline for **Ash Hawk**. It extends the current evaluation → lesson generation → lesson application loop into a more disciplined **research and promotion system** for improving agent policy, skills, tools, harnesses, and evaluation quality.

The pipeline is designed to do five things well:

1. understand failures
2. route failures to the correct improvement surface
3. generate and apply targeted changes
4. verify durable improvement
5. promote reusable learning while preventing regression

---

## 2. Goals

The pipeline should:

* improve agent performance through iterative cycles
* avoid overfitting to narrow eval packs
* separate behavioral fixes from infrastructure fixes
* classify failures before proposing changes
* support deterministic and LLM-assisted evaluation
* preserve experiment history and lesson lineage
* allow promotion, rollback, demotion, and retirement of lessons
* generate reusable knowledge from repeated wins and losses

---

## 3. Non-Goals

This pipeline is not intended to:

* blindly maximize a single score
* auto-promote changes without evidence
* treat all failures as policy problems
* replace human oversight for high-risk production promotion
* assume lessons that work for one agent should apply globally

---

## 4. High-Level Architecture

```text
┌──────────────────────────────────────────────────────────────────────┐
│                        ASH HAWK IMPROVEMENT LOOP                    │
└──────────────────────────────────────────────────────────────────────┘

Baseline / Failed Run
        │
        ▼
 Competitor (optional replay)
        │
        ▼
 Translator
        │
        ▼
 Analyst
        │
        ▼
 Triage
   ┌────┴────┐
   ▼         ▼
Coach    Architect
   └────┬────┘
        ▼
 Curator
        ▼
 Experiment Designer
        ▼
 Applier
        ▼
 Verifier
        ▼
 Promotion Manager
        │
        ├── promoted lessons ───────────────▶ Librarian
        └── run metadata / experiment data ─▶ Historian

Parallel hardening loop:
Adversary ──▶ new adversarial scenarios / eval additions
```

## 5. Pipeline Stages

### 5.1 Baseline / Failed Run

The system begins with either:

* a baseline evaluation run of the current agent
* a failed scenario from prior experiments
* a regression detected by the verifier
* an adversarial failure generated by the adversary
* a human-submitted problem case

This produces the raw artifact bundle that seeds the improvement loop.

#### Inputs

* agent identifier
* experiment identifier
* eval pack
* scenario path or scenario bundle
* run transcript
* model/tool traces
* prior lessons active during the run

#### Outputs

* `RunArtifactBundle`

### 5.2 Competitor

The Competitor replays failed or weak runs with candidate lessons or alternate execution strategies applied. Its job is not to author final lessons, but to answer:

* did replaying with targeted changes help?
* what changed between baseline and replay?
* which lessons appear promising enough for analysis?

This role is optional, but strongly recommended when replay is cheap enough.

#### Responsibilities

* rerun failed tasks under controlled conditions
* compare baseline vs replay
* isolate promising changes
* generate structured comparison evidence

#### Inputs

* `RunArtifactBundle`
* candidate lesson set
* replay controls
* experiment constraints

#### Outputs

* `CompetitorOutput`

#### Success Criteria

* baseline and replay are comparable
* improvement deltas are measurable
* evidence is sufficient for downstream use

### 5.3 Translator

The Translator normalizes upstream artifacts into a structured schema that downstream agents can reliably consume.

It is responsible for converting raw run details into standardized findings and strategy mappings.

#### Responsibilities

* validate schema
* normalize artifacts
* map findings to strategies and sub-strategies
* enrich artifacts with canonical metadata
* reject malformed or weak evidence

#### Inputs

* `RunArtifactBundle`
* `CompetitorOutput` if present

#### Outputs

* `TranslatorOutput`

#### Success Criteria

* downstream roles receive consistent structured objects
* strategy mapping is explicit
* malformed inputs are surfaced early

### 5.4 Analyst

The Analyst performs deep inspection of the run and generates root-cause findings.

#### Responsibilities

* identify patterns of failure
* calculate metrics
* inspect tool usage efficiency
* inspect transcript behavior
* identify risk areas
* identify recurring weaknesses across runs

#### Inputs

* `TranslatorOutput`
* prior experiment summaries
* historical metrics if available

#### Outputs

* `AnalystOutput`

#### Success Criteria

* findings contain severity and evidence
* patterns are grouped meaningfully
* analysis supports triage routing

### 5.5 Triage

Triage is the routing brain of the system. It decides what kind of failure occurred and where the fix belongs.

#### Why Triage Exists

Without triage, Coach and Architect may both propose fixes for the same symptom. Triage forces the system to identify the correct improvement surface.

#### Failure Categories

At minimum:

* `POLICY_ORDERING`
* `POLICY_GUARDRAIL`
* `SKILL_MISSING`
* `SKILL_WEAK`
* `TOOL_MISSING`
* `TOOL_INTERFACE_POOR`
* `TOOL_OBSERVABILITY_POOR`
* `HARNESS_LIMITATION`
* `EVAL_GAP`
* `PLANNER_FAILURE`
* `MEMORY_CONTEXT_FAILURE`
* `RETRIEVAL_FAILURE`
* `NONDETERMINISM`
* `ENVIRONMENTAL_FLAKE`
* `MULTI_CAUSAL`

#### Responsibilities

* classify failure type
* assign primary and secondary causes
* route to Coach, Architect, or both
* provide confidence and evidence
* identify cases that should be blocked rather than improved

#### Inputs

* `AnalystOutput`

#### Outputs

* `TriageOutput`

#### Success Criteria

* routing is explicit
* cause confidence is recorded
* multi-causal cases are supported

### 5.6 Coach

Coach focuses on **agent behavior improvements**.

#### Coach Owns

* policy changes
* playbook improvements
* skill creation or refinement
* step-ordering guidance
* reasoning and interaction patterns
* reusable behavioral procedures

#### Responsibilities

* turn failure patterns into policy/skill proposals
* cluster issues by strategy
* generate proposals with evidence
* recommend validation experiments

#### Inputs

* `AnalystOutput`
* `TriageOutput`

#### Outputs

* `ImprovementProposal[]`

#### Proposal Types

* `POLICY_PATCH`
* `POLICY_REORDER`
* `PLAYBOOK_UPDATE`
* `SKILL_CREATE`
* `SKILL_REVISE`
* `PROMPT_GUARDRAIL`
* `BEHAVIORAL_CHECKLIST`

#### Success Criteria

* proposals are scoped
* evidence is traceable
* change surface is behavior-oriented

### 5.7 Architect

Architect focuses on **infrastructure improvements**.

#### Architect Owns

* tool changes
* tool wrappers
* harness changes
* eval improvements
* interface contracts
* observability changes
* config and orchestration improvements

#### Responsibilities

* turn analyst and triage findings into infra proposals
* propose harness/tool/eval changes
* isolate changes from behavior proposals
* identify reusable system-level fixes

#### Inputs

* `AnalystOutput`
* `TriageOutput`

#### Outputs

* `ImprovementProposal[]`

#### Proposal Types

* `TOOL_CREATE`
* `TOOL_REVISE`
* `TOOL_WRAPPER_UPDATE`
* `HARNESS_PATCH`
* `EVAL_PATCH`
* `EVAL_EXPANSION`
* `OBSERVABILITY_IMPROVEMENT`
* `CONFIG_ADJUSTMENT`

#### Success Criteria

* proposals are infra-oriented
* evidence supports the change
* risk and rollout considerations are explicit

### 5.8 Curator

Curator is the first quality gate. It does not verify final durable improvement; it decides which proposals are worthy of experimentation.

#### Responsibilities

* deduplicate proposals
* reject low-evidence ideas
* combine compatible proposals
* split proposals that are too broad
* assess risk
* determine readiness for experiment design

#### Inputs

* `ImprovementProposal[]` from Coach
* `ImprovementProposal[]` from Architect

#### Outputs

* `CuratedLesson[]`

#### Quality Gates

Minimum examples:

* evidence required
* confidence threshold met
* risk level allowed
* change surface clearly identified
* rollback path present
* proposal not already retired unless explicitly retried

#### Success Criteria

* only experiment-worthy proposals advance
* duplicates and weak proposals are filtered out
* lesson IDs and lineage are assigned

### 5.9 Experiment Designer

Experiment Designer determines exactly how a curated lesson should be tested.

#### Why It Exists

A policy tweak, a new tool wrapper, and an eval patch should not all be tested the same way.

#### Responsibilities

* choose target scenario sets
* choose trial count
* choose isolation vs bundled testing
* define acceptance criteria
* define rejection criteria
* define rollback criteria
* define required pack coverage
* decide cross-pack or single-pack validation
* specify budget and latency tolerances

#### Inputs

* `CuratedLesson[]`
* current eval packs
* historical run data
* agent metadata

#### Outputs

* `ExperimentPlan[]`

#### Experiment Types

* single lesson isolated
* bundled compatible lessons
* A/B replay
* adversarial replay
* regression suite validation
* repeated-run stability test
* cross-pack generalization test

#### Success Criteria

* experiments are well scoped
* success criteria are explicit
* repeatability requirements are stated

### 5.10 Applier

Applier turns curated lessons and experiment plans into concrete system changes.

#### Responsibilities

* apply lessons to the correct surface
* generate change sets
* track touched files and configs
* stage temporary experiment-only changes
* create rollback plans
* preserve change provenance

#### Inputs

* `CuratedLesson[]`
* `ExperimentPlan[]`

#### Outputs

* `ChangeSet`

#### Change Surfaces

* agent policy file
* skill registry
* playbook definitions
* tool metadata
* tool wrapper implementation
* harness config
* eval pack definitions
* retry thresholds
* planner settings
* observability config

#### Success Criteria

* changes are traceable
* changes are reversible
* experiment scope is respected

### 5.11 Verifier

Verifier determines whether the applied change produced a durable, acceptable improvement.

Curator asks, “is this proposal good enough to test?”
Verifier asks, “did this actually work well enough to keep?”

#### Responsibilities

* run deterministic checks
* run regression suites
* check repeatability across trials
* validate schema/tool correctness
* validate latency and token budgets
* validate safety and policy compliance
* compare before/after metrics
* assess whether gains are stable or noisy

#### Inputs

* `ChangeSet`
* `ExperimentPlan[]`
* baseline metrics
* active eval packs
* regression packs
* adversarial scenarios if required

#### Outputs

* `VerificationReport`

#### Verification Dimensions

* correctness
* score delta
* variance / stability
* regression count
* latency delta
* token delta
* tool-call validity
* groundedness / citation correctness if applicable
* safety / policy conformance

#### Success Criteria

* improvement is measured against defined acceptance criteria
* regressions are surfaced clearly
* unstable wins are not silently promoted

### 5.12 Promotion Manager

Promotion Manager decides whether verified lessons should become part of the stable system.

#### Responsibilities

* promote, hold, demote, or retire lessons
* decide scope of application
* resolve conflicts between promoted lessons
* manage lesson versions
* manage rollout status
* maintain promotion thresholds
* trigger rollback when necessary

#### Inputs

* `VerificationReport`
* lesson history
* current promoted lesson registry
* conflict graph if available

#### Outputs

* `PromotionDecision[]`

#### Decision Types

* `PROMOTE_GLOBAL`
* `PROMOTE_AGENT_SPECIFIC`
* `PROMOTE_PACK_SPECIFIC`
* `HOLD_FOR_MORE_DATA`
* `DEMOTE`
* `RETIRE`
* `ROLLBACK`

#### Success Criteria

* promotions are evidence-based
* scope is explicit
* rollback conditions are preserved

### 5.13 Librarian

Librarian turns promoted learning into reusable knowledge.

#### Responsibilities

* convert promoted lessons into reusable patterns
* maintain anti-pattern registry
* maintain known-good templates
* map lessons to strategies and contexts
* support future proposal generation
* prevent rediscovery of the same lessons

#### Inputs

* `PromotionDecision[]`
* promoted lessons
* verification evidence

#### Outputs

* `KnowledgeEntry[]`

#### Knowledge Types

* policy pattern
* skill template
* tool wrapper pattern
* eval hardening pattern
* anti-pattern
* retry recipe
* failure taxonomy note
* strategy playbook entry

#### Success Criteria

* knowledge is reusable
* applicability conditions are clear
* duplicate rediscovery decreases over time

### 5.14 Historian

Historian owns experiment history and long-range telemetry.

#### Responsibilities

* record experiment lineage
* track proposal performance over time
* surface drift and decay
* identify stale promoted lessons
* identify frequently failing proposal families
* support dashboards and analytics
* preserve audit trail

#### Inputs

* run artifacts
* analyst outputs
* experiment plans
* verification reports
* promotion decisions

#### Outputs

* `ExperimentHistorySummary`
* persistent telemetry records

#### Success Criteria

* every lesson has lineage
* history supports later analysis
* trends are queryable by strategy, agent, and proposal type

### 5.15 Adversary

Adversary generates new stress cases so the system does not simply learn to game the current eval pack.

#### Responsibilities

* generate adversarial scenarios
* target known weak spots
* expand coverage
* create edge-case and trap tasks
* introduce ambiguity, noise, contradictions, and failure pressure
* propose new eval additions

#### Inputs

* historical weaknesses
* promoted lesson history
* verifier regressions
* current eval gaps

#### Outputs

* `AdversarialScenario[]`
* `EvalExpansionProposal[]`

#### Success Criteria

* new scenarios are meaningfully distinct
* scenarios test real blind spots
* overfitting is reduced over time

## 6. Control Flow

### 6.1 Standard Control Flow

```text
1. Collect baseline or failed run
2. Optional replay via Competitor
3. Normalize via Translator
4. Analyze via Analyst
5. Route via Triage
6. Generate proposals via Coach and/or Architect
7. Filter and gate via Curator
8. Design experiments via Experiment Designer
9. Apply changes via Applier
10. Verify outcomes via Verifier
11. Decide promotion via Promotion Manager
12. Store reusable knowledge via Librarian
13. Record telemetry via Historian
14. Feed new weak spots to Adversary
15. Repeat
```

### 6.2 Routing Rules

* If `Triage.primary_owner == COACH`, Architect may be skipped unless secondary cause requires infra change.
* If `Triage.primary_owner == ARCHITECT`, Coach may be skipped unless behavior change is also needed.
* If `MULTI_CAUSAL`, both run, and Curator resolves duplication or bundling.
* If `EVAL_GAP`, route primarily to Architect and Adversary.
* If `NONDETERMINISM`, block promotion until stability is addressed.

## 8. Interfaces by Role

### 8.1 Expanded Role Contract

Every role must expose a runtime contract, prompt pack, and validation surface.

```python
from typing import Generic, TypeVar
from pydantic import BaseModel, Field

TInput = TypeVar("TInput", bound=BaseModel)
TOutput = TypeVar("TOutput", bound=BaseModel)


class RolePromptPack(BaseModel):
    system_prompt_path: str
    task_template_path: str
    rubric_path: str
    example_paths: list[str] = Field(default_factory=list)


class RoleRuntimeConfig(BaseModel):
    model_name: str
    temperature: float = 0.0
    max_output_tokens: int | None = None
    structured_output_required: bool = True
    allowed_tools: list[str] = Field(default_factory=list)


class RoleAgent(Generic[TInput, TOutput]):
    name: str
    mission: str
    prompt_pack: RolePromptPack
    runtime_config: RoleRuntimeConfig

    def run(self, payload: TInput) -> TOutput:
        raise NotImplementedError

    def validate_scope(self, payload: TInput) -> bool:
        raise NotImplementedError

    def validate_output(self, output: TOutput) -> bool:
        raise NotImplementedError
```

### 8.2 Role Registry

Each role should be registrable and composable.

```python
class RoleRegistry:
    def __init__(self) -> None:
        self._roles: dict[str, RoleAgent] = {}

    def register(self, name: str, role: RoleAgent) -> None:
        self._roles[name] = role

    def get(self, name: str) -> RoleAgent:
        return self._roles[name]
```

## 9. Role Prompt and Agent Contract Spec

Every role in the Ash Hawk improvement pipeline must have an explicitly defined **agent contract**. A role is not considered complete if it only has a name and a place in the pipeline. Each role must define its purpose, allowed behavior, boundaries, prompt assets, schema, and runtime configuration.

This prevents role drift, cross-role contamination, vague outputs, and duplicated reasoning across the pipeline.

### 9.1 Why This Exists

Without explicit contracts, roles tend to collapse into generic LLM calls that:

* produce overlapping outputs
* infer responsibilities that belong to other roles
* make unsupported claims
* mix analysis, proposal, and approval work together
* become difficult to evaluate or replace

The purpose of the contract is to make each role:

* distinct
* testable
* composable
* replaceable
* measurable

### 9.2 Required Definition for Every Role

Each role must define the following:

1. **Role Name**
   A stable name used by orchestration, telemetry, storage, and prompt loading.

2. **Mission**
   A short statement of what the role is responsible for.

3. **Allowed Actions**
   The kinds of conclusions or outputs the role is allowed to produce.

4. **Forbidden Actions**
   The kinds of outputs or decisions the role must not produce.

5. **Required Inputs**
   The exact objects and context the role receives.

6. **Expected Outputs**
   The Pydantic schema or structured output type the role must return.

7. **Decision Rules**
   How the role should handle ambiguity, uncertainty, weak evidence, or multi-causal scenarios.

8. **Quality Bar**
   The rubric used to judge whether the role performed correctly.

9. **Failure Behavior**
   What the role should do when inputs are incomplete, contradictory, malformed, or insufficient.

10. **Tool Access**
    Which tools or services the role may call, if any.

11. **Model Settings**
    Recommended model class, temperature, determinism preferences, and any structured-output requirements.

12. **Prompt Assets**
    Each role must define:

    * a system prompt
    * a task prompt template
    * a rubric document
    * examples of strong outputs
    * examples of weak outputs

Every role should conform to this template:


### 9.4 Prompt Layers

Each role should use three distinct prompt layers.

#### A. System Prompt

Defines the role identity, scope, and behavior. This should remain stable across tasks.

Examples:

* Triage classifies failures and assigns ownership
* Coach proposes behavioral improvements only
* Verifier judges outcomes conservatively and does not invent new proposals

#### B. Task Prompt Template

Defines the run-specific work the role is being asked to perform.

Examples:

* “Analyze this failed run and return a `TriageOutput`”
* “Given these findings, generate up to 3 behavior-focused proposals”
* “Evaluate this change set against the attached experiment plan and return a `VerificationReport`”

#### C. Output Schema

Constrains the role into a predictable structured response.

The schema is not optional. A role without schema-constrained output is considered incomplete.

### 9.6 Role-Specific Prompt Guidance

#### Competitor

**Mission**
Replay failed or weak runs with targeted changes applied and compare baseline versus replay.

**Prompt Focus**

* compare measurable behavior
* isolate replay differences
* avoid over-interpreting results

**Must Not**

* invent new architecture
* promote lessons
* directly author final policy

**Recommended Prompt Style**

* evidence-heavy
* comparison-oriented
* low speculation

#### Translator

**Mission**
Normalize raw run artifacts into canonical structured objects for downstream roles.

**Prompt Focus**

* preserve evidence
* map raw artifacts into schema
* reject malformed inputs
* avoid speculation

**Must Not**

* do root-cause analysis
* author proposals
* assign final ownership

**Recommended Prompt Style**

* highly constrained
* low temperature
* parser-like behavior

#### Analyst

**Mission**
Identify patterns, weaknesses, severity, metrics, and possible causes from structured artifacts.

**Prompt Focus**

* distinguish symptoms from likely causes
* group repeated signals
* calculate or summarize meaningful metrics
* preserve severity discipline

**Must Not**

* approve changes
* author final promotion decisions
* directly rewrite the system

**Recommended Prompt Style**

* reasoning-oriented
* evidence-anchored
* pattern-focused

#### Triage

**Mission**
Classify the failure type and assign the correct owner for improvement.

**Prompt Focus**

* classify before proposing
* identify primary and secondary causes
* prefer uncertainty to false certainty
* use multi-causal classification when appropriate

**Must Not**

* draft final change proposals
* redesign tools directly
* rewrite policy itself

**Recommended Prompt Style**

* highly disciplined
* classification-first
* conservative under ambiguity

#### Coach

**Mission**
Generate behavior-focused proposals such as policy, playbook, skill, and checklist improvements.

**Prompt Focus**

* behavior changes only
* proposals must be scoped and testable
* map failures to reusable behavioral fixes

**Must Not**

* create new tools
* rewrite harness internals
* patch eval packs unless explicitly allowed

**Recommended Prompt Style**

* proposal-oriented
* behavior-scoped
* practical and testable

**Recommended Variants**

* policy proposal mode
* skill proposal mode
* behavioral checklist mode

#### Architect

**Mission**
Generate infrastructure-focused proposals such as tool, harness, eval, interface, and observability improvements.

**Prompt Focus**

* infrastructure changes only
* justify complexity
* prefer minimal sufficient change
* avoid tool proliferation

**Must Not**

* solve every problem with a new tool
* rewrite agent behavior unless triage routes to both
* approve its own changes

**Recommended Prompt Style**

* systems-oriented
* anti-bloat
* evidence-constrained

**Recommended Variants**

* tool proposal mode
* harness proposal mode
* eval proposal mode
* observability proposal mode

#### Curator

**Mission**
Filter, deduplicate, split, combine, and approve or reject proposals for experimentation.

**Prompt Focus**

* proposal readiness
* evidence sufficiency
* scope clarity
* rollback clarity
* duplication control

**Must Not**

* run verification
* promote lessons
* generate broad new proposals unrelated to inputs

**Recommended Prompt Style**

* gatekeeping
* conservative
* structure-oriented

#### Experiment Designer

**Mission**
Design the correct validation plan for a curated lesson.

**Prompt Focus**

* choose isolated or bundled tests
* select scenario sets
* define acceptance and rejection criteria
* define repeat count and coverage requirements

**Must Not**

* apply changes
* promote lessons
* rewrite the proposal itself

**Recommended Prompt Style**

* structured
* experimental-design-focused
* explicit on criteria

#### Applier

**Mission**
Convert a curated lesson and experiment plan into a reversible concrete change set.

**Prompt Focus**

* respect target surface
* preserve reversibility
* keep modifications scoped
* preserve provenance

**Must Not**

* introduce unrelated edits
* broaden experiment scope
* self-approve risky changes

**Recommended Prompt Style**

* operational
* deterministic where possible
* tool-first when supported

#### Verifier

**Mission**
Judge whether the applied changes produced durable improvement against defined criteria.

**Prompt Focus**

* measure outcome against acceptance criteria
* surface regressions
* be skeptical of noisy gains
* recommend reject, hold, or promote

**Must Not**

* invent new lessons
* rewrite production policy
* ignore regressions in favor of score wins

**Recommended Prompt Style**

* conservative
* outcome-centered
* verification-first

**Recommended Variants**

* deterministic verifier
* rubric verifier
* adversarial verifier
* cross-pack verifier

#### Promotion Manager

**Mission**
Decide whether verified lessons should be promoted, held, demoted, retired, or rolled back.

**Prompt Focus**

* lesson lifecycle state
* stability and scope
* conflict checking
* rollout suitability

**Must Not**

* generate new proposals
* perform verification itself
* rewrite changes directly

**Recommended Prompt Style**

* decision-oriented
* scope-aware
* conservative on promotion

#### Librarian

**Mission**
Convert promoted lessons into reusable patterns, anti-patterns, and reference knowledge.

**Prompt Focus**

* reusable knowledge extraction
* applicability conditions
* anti-pattern capture
* de-duplication of rediscovered lessons

**Must Not**

* re-approve lessons
* alter production configuration
* replace historical evidence with summaries only

**Recommended Prompt Style**

* pattern-oriented
* synthesis-focused
* reusable output focused

#### Historian

**Mission**
Capture experiment lineage, trends, drift, stale lessons, and recurring regressions over time.

**Prompt Focus**

* longitudinal insight
* proposal family performance
* trend analysis
* experiment comparison

**Must Not**

* generate direct production proposals unless explicitly requested
* over-index on one run when multi-run trend data exists

**Recommended Prompt Style**

* summary and trend-oriented
* telemetry-aware
* low-drama interpretation

#### Adversary

**Mission**
Generate realistic adversarial scenarios that expose blind spots and overfitting.

**Prompt Focus**

* stress ambiguity
* create contradiction
* expose brittle tool or policy behavior
* expand eval coverage with meaningful hard cases

**Must Not**

* score final outcomes
* promote lessons
* directly patch the system

**Recommended Prompt Style**

* creative but bounded
* realistic
* hardening-oriented

### 9.9 Failure Handling Rules for All Roles

Every role must follow these failure handling rules.

#### Incomplete Evidence

If evidence is incomplete, the role must:

* mark uncertainty explicitly
* avoid unsupported conclusions
* return partial structured output if possible

#### Contradictory Inputs

If inputs conflict, the role must:

* note the contradiction
* preserve competing interpretations where necessary
* avoid flattening conflict into false certainty

#### Schema Failure

If the role cannot produce a valid output:

* return a structured failure object if supported
* or fail fast with explicit error metadata

#### Out-of-Scope Inputs

If the task requests something outside the role’s responsibility:

* refuse that portion
* record the boundary violation
* avoid silently doing another role’s work

## 10. Orchestration Spec

### 10.1 Orchestrator Responsibilities

The orchestrator should:

* manage stage transitions
* persist intermediate outputs
* support checkpointing
* support retries
* route by triage result
* record lineage
* handle partial failures
* enforce promotion rules
* allow optional role disabling

### 10.2 Suggested Orchestrator Flow

```python
class ImproveCycleOrchestrator:
    def run_cycle(self, run_bundle: RunArtifactBundle) -> dict:
        competitor_output = self.maybe_run_competitor(run_bundle)
        translator_output = self.run_translator(run_bundle, competitor_output)
        analyst_output = self.run_analyst(translator_output)
        triage_output = self.run_triage(analyst_output)

        proposals = []
        if triage_output.primary_owner in ("coach", "both"):
            proposals.extend(self.run_coach(analyst_output, triage_output))
        if triage_output.primary_owner in ("architect", "both"):
            proposals.extend(self.run_architect(analyst_output, triage_output))

        curated_lessons = self.run_curator(proposals)
        experiment_plans = self.run_experiment_designer(curated_lessons)

        verification_reports = []
        for plan in experiment_plans:
            change_set = self.run_applier(curated_lessons, plan)
            report = self.run_verifier(change_set, plan)
            verification_reports.append(report)

        promotion_decisions = self.run_promotion_manager(verification_reports)

        self.run_librarian(promotion_decisions, verification_reports)
        self.run_historian(
            run_bundle=run_bundle,
            analyst_output=analyst_output,
            curated_lessons=curated_lessons,
            verification_reports=verification_reports,
            promotion_decisions=promotion_decisions,
        )
        self.run_adversary_if_needed(analyst_output, verification_reports)

        return {
            "triage": triage_output,
            "proposals": proposals,
            "curated_lessons": curated_lessons,
            "verification_reports": verification_reports,
            "promotion_decisions": promotion_decisions,
        }
```

## 12. Quality Gates

### 12.1 Curator Quality Gate

A proposal should generally be rejected if:

* confidence is below threshold
* no evidence is attached
* target surface is ambiguous
* rollback path is missing for medium/high risk changes
* proposal duplicates a retired lesson without new evidence
* proposal bundles unrelated changes
* proposal appears to patch symptoms only

### 12.2 Verifier Quality Gate

A change should fail verification if:

* primary correctness regresses
* score delta is insignificant and noisy
* variance exceeds allowed threshold
* regressions appear in protected packs
* token or latency increase exceeds threshold without corresponding gain
* tool schema breaks
* safety or policy conformance fails

### 12.3 Promotion Gate

A lesson should not be promoted if:

* it only wins on a narrow single pack when cross-pack validation is required
* it introduces unstable gains
* it conflicts with already promoted lessons
* it cannot be scoped safely
* it has a poor rollback path

## 13. Promotion Rules

### 13.1 Promotion Thresholds

Suggested defaults:

* isolated low-risk policy/skill lesson: 3 successful verifications
* medium-risk tool/harness lesson: 5 successful verifications
* eval pack expansion: 2 successful validations plus human review if production-critical
* global promotion: must pass cross-pack validation

Supported scopes:

* global

## 14. Failure Taxonomy

This taxonomy should be first-class in Ash Hawk.

### 14.1 Behavior Failures

* skipped required investigation
* acted too early
* failed to summarize
* ignored constraints
* used poor step order
* failed to compare variants
* weak self-checking

### 14.2 Skill Failures

* missing repeatable process
* existing skill too vague
* skill not triggered
* skill triggered incorrectly
* skill over-applied

### 14.3 Tool Failures

* tool absent
* tool poorly described
* tool output too noisy
* tool lacks structured result
* tool observations insufficient
* wrong tool selected

### 14.4 Harness Failures

* orchestration issue
* retry issue
* timeout issue
* missing metadata
* poor checkpointing
* insufficient isolation

### 14.5 Eval Failures

* checklist too easy to game
* missing adversarial coverage
* missing regression scenario
* scoring misaligned with real quality
* grader disagreement unresolved

### 14.6 Environment Failures

* flaky dependency
* non-reproducible run
* external state mutation
* intermittent tool availability

* ambiguity

* misleading tool results

* partial success traps
* latency pressure

Run adversary when:

* telemetry-rich history
* role-specific prompt contracts and evaluation assets

The essential idea is:

**do not just generate lessons — classify them, test them correctly, verify they generalize, promote them carefully, preserve what was learned, and define each role as a real agent contract rather than a loose label.**
